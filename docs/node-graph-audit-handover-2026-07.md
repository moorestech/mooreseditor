## 未対応: 設計判断が必要（実装前に方針決めが要る）

### 1. キャンバスから削除した research ノードのマスタ依存が凍結される

- `exportToMasterResearchPatch.ts` の `patchResearchColumn` はキャンバス上にノードが存在する
  research レコードしか再計算しない。ノード削除後、そのレコードの `prevResearchNodeGuids` /
  `clearedActions` は旧値のまま残る。「保存時全再構築で必ず整合」の残る破綻点。
- 論点: グラフに居ないレコードを「クリアする」か「触らない」か。部分的にグラフを作る運用
  （グラフ管理外のresearchが存在する）を許すなら無条件クリアはデータ破壊になる。
  例えば「一度でもグラフ管理下に入ったレコードのGUID一覧を nodeGraph.v1.json に持ち、
  その集合に限りノード不在ならクリア」等の設計が必要。

### 2. スキーマフィールド名ハードコードの根治（SchemaMeta の構造的限界）

- 現存する直書き:
  - `spatialUnlock.ts:22-28` — `record?.itemGuid` と schemaId `"blocks"`
  - `exportToMasterResearchPatch.ts` — `prevResearchNodeGuids` / `clearedActions`（書き込み先フィールド名）
  - `exportToMasterHelpers.ts:17-22` — `gameActionType` / `gameActionParam` / `unlockItemGuids` /
    値 `"unlockItemRecipeView"`
  - `recipeEdgeConstants.ts:3-6` — `RECIPE_SCHEMA_MAP`（craftRecipes/machineRecipes 固定マップ）
- これらは「依存関係」「アンロック」等のドメイン意味論であり、現行 SchemaMeta
  （guidField/nameField/dataArrayPath/elementSchema のみ推論）では動的解決**できない**。
  根治は (a) スキーマ側に役割アノテーションを足して SchemaMeta が読む、または
  (b) moorestech 固有マッピングをプラグイン設定層（plugin.json 等）に分離、のどちらか。
  場当たり的な修正はしないこと（CLAUDE.md のハードコード禁止と衝突し続ける）。

### 3. ホスト↔プラグインの API バージョンゲート・共有依存バージョン照合がない

- `plugin-sdk` の `SHARED_DEPENDENCIES` は `versionRange` を宣言するが実行時に一切照合されない。
  HostAPI にも互換レンジ照合がない。事前ビルド済み dist 配布モデルなので、ホスト更新で旧 dist が
  「does not provide an export named …」系の不透明なエラーで死ぬ。
- 推奨: plugin.json に対応ホストAPIレンジを持たせロード時照合＋共有依存はロード時に
  ホスト実バージョンと versionRange を突き合わせて警告。

## 未対応: High（発火時の影響が大きい）

- **guidField が `type:"uuid" && autoGenerated:true` のみ検出**（`schemaMeta.ts:47-51`）
  手動UUIDスキーマでは guidField=null になり、レコード検索・表示名解決・レコード作成・
  PropertiesPanel 等 8箇所以上が**無言で**空返しする。最低限、検出失敗時に console.warn を出すべき。
  nameField 推論も弱い（"name"部分一致→最初のstring。"title"/"label"非対応）。
- **masterRecordCreation がSDKレガシー生成経路**（`masterRecordCreation.ts:85-89` の
  `createInitialValue(..., false)`）。switchフィールド配下が生成されない・循環refで無限再帰リスク・
  array minLength 無視。レシピ作成側（DataInitializer 経路）と挙動が真逆。統一するなら
  既存テスト（masterRecordCreation.test.ts）の期待値も見直しが必要。

## 未対応: Medium

- **保存済みビューポートが復元されない**（`NodeCanvas.tsx` の `defaultViewport` は初回マウント時のみ。
  LOAD_GRAPH 後に反映されず、最初のパンで保存値を上書き喪失）
- **SET_NODES/SET_EDGES がクロージャ state から完成配列を計算**（`useNodeOperations.ts` 等）。
  reducer 側で changes を適用する形にすれば連続操作の更新喪失リスクが消える
- **plugin.json の実体バリデーションなし・plugin id 重複ガードなし**（`loader.ts:197-217` /
  `usePlugins.ts`。重複 id は React key 衝突と dispose 誤作動を起こす）
- **saveProject が非アトミック**（`projectPersistence.ts:158-169`。master と nodeGraph の
  部分書き込みで desync しうる。契約に明記済みの既知仕様だが tmp-write→rename 化が望ましい）
- **loadPlugin の `..` トラバーサル許容**（`loader.ts:157-175`。read/saveExtraFile 側は拒否しており
  同一リポで方針不一致。全体監査の C3「許容」判断に含まれるが、修正コストは低い）

## 未対応: Low（要約）

- ドラッグ中の毎フレーム全ノード名前解決 O(N×M)（`NodeEditorApp.tsx:39-46`）、
  Context 値のメモ化なし、schemaMetas の二重計算（NodeEditorInner と NodeEditorApp）
- `useNodeGraph` の `reload` 戻り値が未配線のデッドコード
- ResearchNode の `key={label}` 重複リスク、AddNodeMenu の検索語リセット漏れ、
  `pathValue.setValueAtPath` の範囲外indexで疎配列
- ノード/エッジ配色のハードコード散在（ダークテーマ非対応、矢印マーカー色 #888 の二重定義）
- PluginView.dispose 未実装（一度表示したプラグインツリーがセッション中解放されない）
- Tauri テストモック（`test/mocks/tauri-api-core.ts`）が invoke のみで実API面と乖離
- 空間アンロックの完全同一座標=未割当・等距離=追加順（テストで意図仕様として固定済み）
