# Plugin SDK 規約準拠ディレクトリ再編 設計

## 目的

`packages/plugin-sdk` に残る次の決定論的な規約違反を、公開 API と実行時挙動を変えずに解消する。

- `components/FormView/Field.test.tsx`: 666 行
- `utils/dataInitializer.test.ts`: 238 行
- `components/FormView`: 直下 18 ファイル
- `schema`: 直下 11 ファイル
- `utils`: 直下 20 ファイル

完了時には、変更対象の TypeScript/TSX ファイルをすべて 200 行以下、変更対象ディレクトリの直下をすべて 10 ファイル以下にする。

## スコープ

対象は `packages/plugin-sdk` の上記 5 件だけとする。アプリ本体、node-graph、既存の `FormView/inputs` など、今回の差分で新たに触れない既存違反は対象外とする。

本作業は構造整理であり、スキーマ種別、初期値生成規則、フォーム描画、入力イベント、エラー表示を変更しない。スキーマは入力 YAML から動的に解釈し、サンプル固有の型名、フィールド名、構造を追加しない。

`dataInitializer.ts` と `primitiveDefaultValue.ts` を `initialization` に含め、`schemaToZod` は `validation`、汎用オブジェクト結合は `object` に分離する。`createInitialValue.ts` と `switchFieldProcessor.ts` は 200 行を超える既存 consumer を移動だけのために変更しないよう直下に維持し、移動後の内部実装を直接参照する。UUID 一意化と外部キー解決のテストは `utils/__tests__` にまとめる。`autoIncrement.ts` と `generateUuid.ts` も直下に維持する。これにより直下の実ファイル数は `utils: 10`、`initialization: 2`、`initialization/__tests__: 3`、`object: 2`、`validation: 2`、`utils/__tests__: 2` となる。

`src/index.ts` の公開シンボルは新しい内部パスから同名で re-export する。パッケージルートから利用する `DataInitializer`、`createInitialValue`、`calculateAutoIncrement`、`generateUuid`、`ensureUniqueAutoGeneratedUuids`、`processSwitchFields`、`deepMerge`、`schemaToZod` の名前と型は変えない。

## 依存関係とデータフロー

再編後もデータフローは変えない。

1. `FormView/index.tsx` が `fields/Field.tsx` にスキーマと値を渡す。
2. `Field.tsx` が runtime schema guard により renderer を選ぶ。
3. renderer が既存 input component または `ArrayField` を描画する。
4. 初期値が必要な経路は `utils/createInitialValue.ts` から `utils/initialization` の既存ロジックを呼ぶ。
5. `onChange` は従来どおり上位へ値を返す。

ファイル移動に伴う相対 import だけを更新する。新しい barrel import は追加せず、内部コードは原則として実体ファイルを直接 import する。

## 互換性とエラー処理

- パッケージルート `packages/plugin-sdk/src/index.ts` の公開 export を維持する。
- `schema/index.ts` の公開 export を維持する。
- 公開されていない内部 deep import の互換 shim は作らない。リポジトリ内の利用箇所は新パスへ更新する。
- 既存の invalid schema 表示、unknown switch fallback、runtime guard の挙動を保持する。
- 例外の握りつぶし、環境判定分岐、サンプルスキーマへの特別扱いは追加しない。

## テストと検証

移動・分割前に対象テストを実行して基準状態を記録する。構造整理では新しい振る舞いを追加しないため、TDD の回帰保護として、各まとまりの移動後に対応テストを実行し、既存ケースが同じ結果になることを確認する。

最終検証は次を行う。

1. plugin-sdk の対象テスト
2. plugin-sdk の全テスト
3. リポジトリ全体の `pnpm run lint`
4. リポジトリ全体の `pnpm run build`
5. 決定論チェックを再実行し、対象 5 件が消えていることを確認

挙動変更がないため、ブラウザ E2E は必須条件としない。ユニットテスト、型検査を含むビルド、lint で import と公開 API の整合性を検証する。

## 非目標

- 既存テスト内容の削減や書き換えによる短縮
- runtime schema 判定や初期値生成の仕様変更
- `FormView/inputs` を含む今回の対象外ディレクトリの追加整理
- アプリ本体または node-graph の規約違反修正
- 旧内部 deep import パスの互換ファイル追加
