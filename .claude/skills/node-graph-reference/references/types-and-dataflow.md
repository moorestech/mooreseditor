## 三つのデータソースの関係

```
┌──────────────────────┐
│  App.tsx             │
│  jsonData: Column[]  │◄──── FormView/TableViewからも編集可能
│  (マスタデータ所有者)  │
└──────┬───────────────┘
       │ props.jsonData（参照）
       │ props.setJsonData（変更）
       ▼
┌──────────────────────┐     保存時     ┌─────────────────────┐
│  NodeEditorContext   │────────────────►│ nodeGraph.v1.json   │
│  (React Flow状態)    │                │ (永続化ファイル)      │
│  nodes, edges,       │◄───────────────│ nodes, edges,       │
│  viewport, isDirty   │    読み込み時   │ viewport             │
└──────────────────────┘                └─────────────────────┘
```

**重要な非対称性**: jsonDataはApp.tsxが所有し、ノードエディタは借り物として扱う。ノードエディタがjsonDataを変更するのは以下のケースのみ:

- レシピエッジ作成時に新規レシピレコードを追加
- レシピエッジ/ノード削除時に孤立レシピレコードを削除
- 保存時にresearch column（prevResearchNodeGuids, clearedActions）をパッチ

## 永続化される型

`types/nodeGraph.ts`。`.mooreseditor/nodeGraph.v1.json`の構造。

### NodeGraphFile

```typescript
interface NodeGraphFile {
  version: 1;
  viewport: { x: number; y: number; zoom: number };
  nodes: GraphNode[]; // 5種類のユニオン
  edges: GraphEdge[]; // 5種類のユニオン
}
```

### ノード — masterGuidの有無が分岐点

- **masterGuid有り**（item, block, research）: マスタデータへの参照。表示名は保存しない
- **masterGuid無し**（note, placeholder）: 自己完結。textフィールドにインラインデータを保持

### エッジ — edgeTypeとtypeの区別

**edgeType**（永続化）: ドメインレベルの分類。5種: `recipe`, `craftRecipe`, `machineRecipe`, `dependency`, `visual`
**type**（ランタイム専用）: React Flowのレンダリングコンポーネントキー。2種: `"recipe"`, `"arrow"`

```
edgeType "recipe"|"craftRecipe"|"machineRecipe" → type "recipe" (RecipeEdge)
edgeType "dependency"|"visual"                  → type "arrow"  (DependencyEdge)
```

### ConnectionDecision

エッジ作成時にEdgeTypeDialogから返される判断結果:

```typescript
type ConnectionDecision =
  | { edgeType: "dependency" | "visual" }
  | { edgeType: "recipe"; recipeRefs: RecipeReference[] };
```

## ランタイム専用データ

React Flowのnode.dataとedge.dataには、永続化されないランタイム情報が含まれる。

### ノードのdata

| フィールド      | 永続化                          | 解決タイミング                          |
| --------------- | ------------------------------- | --------------------------------------- |
| `masterGuid`    | Yes（buildNodeGraphFileで使用） | 読み込み時                              |
| `text`          | Yes（note/placeholder）         | 読み込み時                              |
| `displayName`   | **No**                          | 毎レンダリング（`resolveDisplayNames`） |
| `consumeLabels` | **No**                          | 毎レンダリング（research nodeのみ）     |

### エッジのdata

| フィールド     | 永続化                 | 解決タイミング                              |
| -------------- | ---------------------- | ------------------------------------------- |
| `edgeType`     | Yes                    | 読み込み時                                  |
| `recipeRefs`   | Yes（recipe edgeのみ） | 読み込み時                                  |
| `recipeLabels` | **No**                 | 毎レンダリング（`resolveEdgeRecipeLabels`） |

**なぜ毎レンダリング解決するのか**: マスタデータ側でアイテム名が変更されても、グラフのノードが古い名前を表示し続ける問題を防ぐため。常にjsonDataから最新の名前を取得する。

## SchemaMeta

スキーマ構造のハードコードを防ぐための動的メタ情報。`buildSchemaMetaMap(schemas)`でuseMemoにより一度だけ計算。

```typescript
interface SchemaMeta {
  schemaId: string; // "items", "blocks", etc.
  guidField: string | null; // type:"uuid" + autoGenerated:true のフィールド
  nameField: string | null; // "name"を含むstringフィールド
  dataArrayPath: string; // ルートオブジェクトの配列キー
  elementSchema: ObjectSchema | null;
}
```

### 検出ロジックの限界（これを知らないとハマる）

- **guidField**: `type:"uuid" && autoGenerated:true`でのみ検出。`autoGenerated`がないUUIDフィールドは検出されず、そのスキーマ全体のレコード検索が無効になる
- **nameField**: "name"部分文字列マッチ → 最初のstringフィールドフォールバック。"title"や"label"は優先されない
- **dataArrayPath**: **最初の**array propertyを採用。複数arrayがあるスキーマでは意図しないarrayが選ばれる
- **ネスト非対応**: `root.category.items[]`のようなネストされたarrayは検出できない

## NodeEditorContext

`useReducer`ベースのグローバル状態。dispatchアクションの副作用を正確に理解する必要がある。

### isDirtyの挙動に注意

| Action             | isDirty                              |
| ------------------ | ------------------------------------ |
| `SET_NODES`        | **true**に設定（ドラッグ移動でも！） |
| `SET_EDGES`        | **true**に設定                       |
| `SET_VIEWPORT`     | 変更しない                           |
| `UPDATE_NODE_DATA` | **true**に設定                       |
| `LOAD_GRAPH`       | **false**に設定                      |
| `SET_DIRTY`        | 明示的に指定値                       |

`SET_NODES`がドラッグ移動でもdirtyにするのは`onNodesChange → applyNodeChanges → setNodes`の流れによる。これがonMarkDirty過剰発火の根本原因。

## データフロー図

### 読み込み

```
useNodeGraph
  ├─ Tauri FS: readTextFile("nodeGraph.v1.json")
  ├─ Fallback: fetch() in dev
  ├─ Last resort: importResearchFromMaster()
  └─ validateAndMigrate() → dispatch(LOAD_GRAPH)
```

### レンダリング（useMemoで毎回）

```
resolveDisplayNames(state.nodes, jsonData, schemaMetas)
  → jsonDataからmasterGuidでレコード検索 → nameField取得 → displayName注入

resolveEdgeRecipeLabels(state.edges, jsonData, schemaMetas)
  → recipeRefsからレシピレコード取得 → 正規表現でフィールド分類 → ラベル生成
```

### 保存

```
exportAndSave()
  1. exportToMaster()
     ├─ validateGraph()           — 孤立エッジ除去
     ├─ calculateUnlockedItems()  — 空間割当
     ├─ buildResearchDependencyMap()
     ├─ patchResearchColumn()     — masterデータ書き込み
     └─ buildNodeGraphFile()      — グラフシリアライズ
  2. setJsonData(updatedColumns)  — ★onRequestSaveの前
  3. await onRequestSave()        — ディスク書き込み
  4. dispatch(SET_DIRTY(false))   — 成功時のみ
```
